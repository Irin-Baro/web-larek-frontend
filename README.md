## 🏗️ Архитектура <a id=mvp></a>

### **1. MVP (Model-View-Presenter)**
Проект реализован с использованием архитектурного паттерна MVP, где каждый слой выполняет строго определенную роль:

**Model (Слой данных)** — классы `ProductData`, `CartItem` и `OrderData`. Отвечают за хранение состояния приложения, работу с данными и их валидацию. Модели не имеют доступа к интерфейсу и уведомляют о изменениях через события.

**View (Слой представления)** — UI-компоненты (`Page`, `Card`, `Basket`, `OrderForm`), наследуемые от базового класса `Component`. Отвечают за отрисовку данных на экране и генерацию событий при взаимодействии пользователя с интерфейсом.

**Presenter (Слой управления)** — файл `src/index.ts`. Выступает «дирижером» приложения: создает экземпляры моделей и компонентов, подписывается на события через `EventEmitter` и синхронизирует состояние данных с их отображением.

### Принципы взаимодействия:

**Слабая связанность:** View и Model ничего не знают о существовании друг друга.

**Событийная модель:** Все взаимодействия происходят через брокер событий (`EventEmitter`).

**Единый источник истины:** Данные хранятся только в Моделях, Представления лишь отображают их текущее состояние через метод `render`.

**Изоляция логики:** Бизнес-логика (например, расчет суммы корзины или валидация адреса) инкапсулирована внутри соответствующих моделей.

## 📖 Описание<a id=description></a>

## 1. Базовые классы (Base Classes)

### Класс Api
**Расположение:** `src/components/base/api.ts`  
**Назначение:** Взаимодействие с REST API. Обеспечивает стандартизацию запросов и единообразную обработку ответов сервера.

**Свойства:**
- `baseUrl: string` — базовый адрес сервера
- `options: RequestInit` — глобальные настройки запроса (заголовки)

**Методы:**
- `protected handleResponse(response: Response): Promise<object>` — проверяет статус ответа. Если `response.ok`, возвращает JSON, иначе формирует объект ошибки
- `get(uri: string)` — выполняет GET-запрос к эндпоинту
- `post(uri: string, data: object, method: ApiPostMethods = 'POST')` — выполняет запрос с передачей данных в теле (JSON). Поддерживает методы POST, PUT, DELETE

### Класс EventEmitter
**Расположение:** `src/components/base/events.ts`  
**Назначение:** Брокер событий, реализующий паттерн «Наблюдатель». Позволяет компонентам (View) и моделям (Model) взаимодействовать, не имея прямых ссылок друг на друга.

**Свойства:**
- `_events: Map<EventName, Set<Subscriber>>` — хранилище обработчиков для каждого события

**Методы:**
- `on<T>(eventName: string, callback: (data: T) => void)` — установка слушателя на событие
- `emit<T>(eventName: string, data?: T)` — инициация события с передачей данных
- `trigger<T>(eventName: string, context?: Partial<T>)` — возвращает функцию-коллбэк, генерирующую событие при вызове (удобно для UI-компонентов)
- `off` / `onAll` / `offAll` — методы для управления жизненным циклом подписок

### Класс Component
**Расположение:** `src/components/base/Component.ts`  
**Назначение:** Абстрактный базовый класс для всех компонентов представления. Инкапсулирует общую логику работы с DOM-деревом и предоставляет инструментарий для UI-наследников.

**Конструктор:** Принимает корневой DOM-элемент (`container: HTMLElement`)

**Методы:**
- `toggleClass(element: HTMLElement, className: string, force?: boolean)` — переключение CSS-класса
- `setText(element: HTMLElement, value: string)` — безопасная установка текстового контента
- `setImage(element: HTMLImageElement, src: string, alt?: string)` — установка изображения и альтернативного текста
- `setDisabled(element: HTMLElement, state: boolean)` — управление атрибутом `disabled`
- `render(data?: Partial<T>): HTMLElement` — основной метод отрисовки: заполняет свойства компонента данными из объекта `data` и возвращает корневой элемент

### Класс Model
**Расположение:** `src/components/base/Model.ts`  
**Назначение:** Абстрактный базовый класс для моделей данных. Позволяет отличить простые объекты данных от моделей, обладающих логикой.

**Конструктор:** Принимает начальные данные модели и экземпляр `events` для уведомления о изменениях

**Методы:**
- `emitChanges(event: string, payload?: object)` — сообщает всем подписчикам, что модель изменилась

## 2. Слой коммуникации с сервером (API / Service)

### Класс LarekApi (наследует Api)
**Назначение:** Реализует конкретные методы для работы с бэкендом магазина.

**Методы:**
- `getProducts(): Promise<IProduct[]>` — получает список всех товаров с сервера
- `orderProducts(order: IOrder): Promise<IOrderResult>` — отправляет данные заказа на сервер и получает результат оплаты

## 3. Слой моделей данных (Model)

В этой группе описаны классы, которые хранят состояние приложения. Они работают только с данными и уведомляют систему об изменениях через события.

### Класс CartItem (Модель корзины)
**Наследование:** Наследует базовый класс `Model`

**Назначение:** Управляет списком выбранных товаров и отвечает за расчет стоимости

**Данные:**
- `private items: IProduct[]` — список товаров, добавленных в корзину

**Методы:**
- `add(item: IProduct): void` — добавляет товар в корзину
- `remove(id: string): void` — удаляет товар по ID
- `clear(): void` — полностью очищает корзину (после покупки)
- `getTotal(): number` — возвращает общую сумму всех товаров в корзине
- `getCount(): number` — количество товаров (для иконки на главной)

**События:** `basket:changed` (генерируется при любом изменении состава)

### Класс OrderData (Модель данных заказа)
**Наследование:** Наследует базовый класс `Model`

**Назначение:** Хранит информацию о покупателе и проверяет её корректность. Модель `OrderData` не знает о существовании форм и не имеет методов для валидации отдельных полей. Она предоставляет единый метод `validateAll()`, который проверяет все данные целиком. Задача презентера — решить, какие ошибки из общего списка показать пользователю в зависимости от того, какая форма сейчас открыта.

**Данные:**
- `payment: string` — способ оплаты
- `address: string` — адрес доставки
- `email: string` — почта
- `phone: string` — телефон
- `formErrors: FormErrors` — объект с текстами ошибок валидации

**Методы:**
- `setField(field: keyof IOrder, value: string): void` — записывает данные из полей ввода
- `validateAll(): FormErrors` — валидирует все поля заказа и возвращает объект с ошибками для каждого поля

**События:** `formErrors:changed` (при появлении/исчезновении ошибок)

## 4. Слой представления (View)

### Класс Page
**Назначение:** Управляет основными элементами на главной странице.

**Элементы:**
- Иконка корзины (счетчик)
- Контейнер для карточек товаров
- Обертка страницы (для блокировки прокрутки)

**Методы:**
- `set counter(value: number)` — обновляет цифру на иконке корзины
- `set catalog(items: HTMLElement[])` — заменяет содержимое витрины на новые карточки
- `set locked(value: boolean)` — блокирует прокрутку страницы при открытии модального окна

### Класс Modal
**Назначение:** Универсальное модальное окно. Служит контейнером для других компонентов (корзины, карточки, форм).

**Методы:**
- `open() / close()` — управление видимостью
- `set content(value: HTMLElement)` — вставляет нужный контент внутрь окна

**События:** `modal:open`, `modal:close`

### Класс Card
**Назначение:** Компонент для отображения товара. Использует один шаблон, но имеет три модификации (через разные наборы данных).

**Элементы:**
- Заголовок
- Цена
- Изображение
- Категория
- Кнопка
- Описание

**Методы:**
- `set price(value: number | null)` — устанавливает цену (или текст «Бесценно»)
- `set category(value: string)` — устанавливает текст и нужный CSS-класс для цвета тега
- `set title(value: string)` — обновляет название товара;
- `set image(value: string)` — обновляет изображение товара;
- `setCartStatus(inCart: boolean): void` — обновляет визуальное состояние карточки, если товар в корзине;
- `setSelected(selected: boolean): void` — выделение карточки.

**События:** `click` (вызывает `card:select`)

### Класс Basket
**Назначение:** Отображает содержимое корзины в модальном окне.

**Элементы:**
- Список товаров
- Итоговая сумма
- Кнопка оформления

**Методы:**
- `set items(items: HTMLElement[])` — отрисовывает список товаров в корзине
- `set total(value: number)` — обновляет итоговую цену

### Класс Form
**Назначение:** Базовый класс для всех форм.

**Элементы:**
- Кнопка сабмита
- Контейнер для текста ошибок

**Методы:**
- `onInputChange(field: keyof T, value: string)` — генерирует событие изменения конкретного поля
- `set valid(value: boolean)` — управляет доступностью кнопки «Далее/Оплатить»
- `set errors(value: string)` — выводит текст ошибки валидации

### Класс OrderForm (Форма доставки)
**Наследование:** Наследует `Form`

**Назначение:** Содержит выбор способа оплаты (кнопки) и поле адреса.

**Метод:**
- `toggleButtons(name: string)` — подсвечивает выбранную кнопку способа оплаты («Онлайн» или «При получении»)

**События:**
- `order:submit` — при переходе к контактам
- `payment:toggle` — генерируется при клике на кнопку выбора способа оплаты

### Класс Contacts (Форма контактов)
**Наследование:** Наследует `Form`

**Назначение:** Содержит поля email и phone.

**События:** `contacts:submit` (финальная отправка заказа)

### Класс Success
**Назначение:** Окно успешного завершения.

**Элементы:**
- Текст с итоговой суммой списания

**События:** `success:close` (возврат к покупкам)

## 5. Слой Презентера (Presenter)

### Роль и расположение
Управление логикой приложения

### Архитектура взаимодействия
Взаимодействие слоев осуществляется через брокер событий (`EventEmitter`):

#### 1. Инициализация приложения:
- Создается экземпляр `EventEmitter` — брокер событий для взаимодействия слоев
- Создается экземпляр `LarekApi` для работы с сервером
- Инициализируются модели данных: `CatalogData` (список товаров), `CartItem` (корзина) и `OrderData` (заказ)
- В слой представления передаются корневые элементы шаблонов

#### 2. Обработка событий моделей (реакция на изменение данных):
- **`items:changed`** — при получении данных от API презентер перерисовывает витрину на главной странице через компонент `Page`
- **`basket:changed`** — при изменении корзины презентер:
  - Обновляет счетчик в шапке страницы
  - Пересчитывает итоговую сумму в модальном окне корзины
- **`formErrors:changed`** — при изменении состояния валидации в модели `OrderData` презентер:
  - Получает полный объект ошибок от модели через `validateAll()`
  - Выбирает из него только те ошибки, которые относятся к текущей открытой форме
  - Передает отфильтрованные ошибки в компонент формы для отображения
  - Блокирует или разблокирует кнопку отправки в зависимости от наличия ошибок

#### 3. Обработка событий представления (реакция на действия пользователя):
- **`card:select`** — при клике на товар презентер:
  - Запрашивает детализацию у модели
  - Открывает модальное окно с описанием товара
- **`card:toBasket`** — презентер вызывает метод добавления товара в модель корзины `CartItem`
- **`basket:open`** — открытие модального окна со списком выбранных товаров
- **`order:open`** — инициирует начало процесса оформления заказа (открытие формы доставки)
- **`order:submit`** — сохранение данных доставки в модель `OrderData` и переход к форме контактов
- **`contacts:submit`** — отправка собранных данных:
  - Из модели `OrderData`
  - Из корзины `CartItem` 
  - На сервер через `LarekApi`

#### 4. Завершение заказа:
После успешного ответа от сервера презентер:
- Инициирует очистку корзины в модели `CartItem`
- Сбрасывает формы в модели `OrderData`
- Показывает пользователю компонент `Success` (сообщение об успехе)

    ```bash
        http://localhost:8080
    ```

    ```bash
    npm run build
    ```
    или

    ```bash
    yarn build
    ```

## 🎯 События приложения<a id=app></a>

Взаимодействие слоев Model и View осуществляется через события, брокером которых выступает `EventEmitter`. События разделены на две группы: события изменения данных (Model) и пользовательские интерфейсные события (View).

### События моделей (Model Events)
Эти события генерируются классами данных, когда в них меняется состояние. На них подписывается Презентер для обновления интерфейса.

- **`items:changed`** — Изменение списка товаров в каталоге (например, после загрузки с сервера). По этому событию обновляется витрина на главной странице.
- **`basket:changed`** — Любое изменение в корзине (добавление или удаление товара). Инициирует обновление счетчика на иконке корзины и перерисовку списка в самой корзине.
- **`formErrors:changed`** — Изменение состояния валидации в модели `OrderData`. Передает полный объект с ошибками для всех полей заказа. Презентер на основе этих данных и информации о текущей открытой форме решает, какие ошибки показать пользователю и доступна ли кнопка отправки.
- **`order:success`** — Событие, которое может быть сгенерировано для уведомления об успешной отправке заказа. В текущей реализации логика очистки корзины, сброса форм и показа окна успеха выполняется непосредственно в обработчике `contacts:submit`.

### События представления (View Events)
Эти события генерируются компонентами интерфейса при взаимодействии пользователя с кнопками, формами и карточками.

- **`card:select`** — Выбор карточки товара на витрине. Передает ID карточки для открытия предпросмотра.
- **`card:toBasket`** — Нажатие кнопки «Купить» на карточке товара. Передает данные товара для добавления в модель корзины.
- **`basket:open`** — Клик по иконке корзины. Инициирует открытие модального окна корзины.
- **`basket:delete`** — Клик по иконке «удалить» в списке товаров корзины. Передает ID для удаления из модели.
- **`order:open`** — Нажатие кнопки «Оформить» в корзине. Открывает первую форму заказа.
- **`order:submit`** — Сабмит первой формы (доставка). Переводит пользователя ко второй форме (контакты).
- **`contacts:submit`** — Финальный сабмит форм. Инициирует отправку данных через `LarekApi`.
- **`input:change`** — Ввод данных в любое поле формы. Позволяет Презентеру мгновенно передавать данные в Модель для валидации.
- **`modal:open` / `modal:close`** — События открытия и закрытия модальных окон (используются для блокировки прокрутки страницы).

## 🔄 Поток данных<a id=data></a>

### 1. Инициализация и загрузка:
- `index.ts` создает экземпляры `LarekApi`, `EventEmitter` и моделей данных (`ProductData`, `CartItem`, `OrderData`)
- `LarekApi` загружает список товаров
- `index.ts` передает полученный список в модель `ProductData`
- Модель `ProductData` генерирует событие `items:changed`

### 2. Взаимодействие пользователя (на примере покупки):
- Пользователь кликает по карточке — компонент `Card` генерирует событие `card:select`
- Презентер ловит событие, открывает `Modal` и отрисовывает в нем превью товара
- При нажатии «В корзину» генерируется событие `card:toBasket`
- Презентер вызывает метод `add()` в модели `CartItem`
- Модель `CartItem` обновляет данные и генерирует событие `basket:changed`
- Презентер ловит `basket:changed` и обновляет счетчик в `Page` и содержимое в компоненте `Basket`

### 3. Оформление заказа:
- Пользователь нажимает «Оформить» — открывается `OrderForm`
- При вводе данных `OrderForm` генерирует `input:change`
- Презентер записывает данные в модель `OrderData` через `setField()`
- Модель `OrderData` валидирует все поля методом `validateAll()` и генерирует событие `formErrors:changed` с полным объектом ошибок
- Презентер получает все ошибки, фильтрует их для текущей формы и обновляет интерфейс
- Кнопка «Далее»/«Оплатить» активна только при отсутствии ошибок валидации
- При финальном подтверждении Презентер собирает данные из `CartItem` и `OrderData`, отправляет их через `LarekApi` и при успехе вызывает очистку всех моделей и показ окна `Success`
